Analyzing Quantum Programs

Learn about the process and idea behind analyzing quantum programs written in pyQuil.

Written by Aaron Vontell on July 26, 2017

As quantum computing becomes a reality, companies have been working hard to create useful and easy-to-use platforms for developing quantum programs. One of these platforms, Forest, includes Quil and pyQuil, developed by Rigetti Computing. Although many other quantum programming frameworks exist, having the honor of working at Rigetti during a summer internship has allowed me to learn a tremendous amount about this specific platform.

Part of my task over the summer of 2017 was to implement pre-existing quantum algorithms using the Quil ecosystem; for example, my friend Harley and I implemented Shor's algorithm using pyQuil. pyQuil is essentially a Python wrapper for Quil, which is a quantum instruction language used to talk to the quantum computers at Rigetti Computing. The wrapper allows us to construct algorithms in a faster and more efficient way, considering the repetition in many of these algorithms; for example, we don't want to write instructions line by line to apply a Hadamard to each qubit. Instead, pyQuil let's us use for-loops and the like to easily make these programs.

However, this automated program writing is something to be wary of; how do you know that your gate and qubit count is within the bounds set by whichever paper you are implementing from? How do you detect bugs where you program has tons of extra, useless gates, but still outputs the correct answer? These sorts of questions inspired me to write some analytical tools for pyQuil. To my surprise, not only was this little project fun, but it actually helped me find bugs in some of the existing quantum programs.

Within the pyQuil codebase, I built a few methods which can basically be broken down into three categories: counting, plotting, and analysis. For the counting methods, I wanted to be able to determine the number of gates, the types of gates, number of qubits, and number of instructions within a certain Quil program. Using these numbers, my analysis methods take in a Python function which outputs a Quil progam, a list of inputs to construct the program with, and a couple other arguments for determining how to deal with the list of inputs. Finally, I have a plotting function which plots relevant stats, where "relevant" basically means any stat which is non-zero, or any stats that the user requests specifically. The plots are made with the awesome and beatiful plotting library Plotly.

I tested this out with our implementation of the QFT, and the results are below:

Now these are actually some cool results! I can see the growth of CPHASE gates, as well as the time it took to construct the Quil program (this is what I mean by compile time). What's even cooler is that I plotted the theoretical bound from a paper regarding the growth of these gates with respect to the number of qubits being operated on.


From this analysis I can see that the number of instructions for my program is growing faster than the theoretical projection for the QFT; this may be due to using a different gate set, or it may be due to larger constant factors in this lower regime of qubits. It may even be a bug with the QFT program; something to look into!

In fact, now that I had this ability to construct these programs, I tried running the QFT on up to 2048 qubits, rather than 0 through about 200 qubits. However, my program crashed! Apparently the code used to implement the QFT Quil program has a recursion depth limit, causing the execution stack to become full. Another bug found! We would have never originally found this bug in the program, because why would we be running the QFT on that many qubits in today's quantum simulators or quantum computers? With these analytical tools, we can begin preparing for future programs which can run on larger qubits.

The use of this software doesn't stop with the QFT; it can be used with any Quil program generator! In fact, we used it to find a few bugs in a friend's code, whose gate counts approached numbers in the billions without him even realizing, with only 50 qubits! My friend then took another look at the code, found the bug, and squashed it.

Unfortunately, this specific analytical tool never got merged into pyQuil; it was a prototype that I spent only a few days on, and the code was quite messy. Hopefully some day soon I get a chance to clean it up, but feel free to check out the original pull request. I would like to implemented more features, such as an interface that Quil program creators can follow to ensure that they program can be easily passed into the analytical tool. Another feature I started to work on was using NumPy's polyfit function to determine the gate and qubit complexity of a quantum program, for further comparison with theoretical bounds.

All said and done, even as a prototype, I now realize that quantum program analytical tools are pretty important; we are developing low level software, and with these kinds of tools, we can find bugs which would otherwise prohibit our programs from running on larger quantum machines.

The Mistake of Universal Mobile Design

Learn why designing applications for multiple platforms requires going back to the drawing board.

Written by Aaron Vontell on Sep 16, 2016

There is a major problem today with the way that mobile applications are designed. Tell me, how many times have you found yourself in the following situation:

You recently picked up some contractual work with a small time company to create an iOS application. Your point of contact debriefs you on the project and functionality of the app, and all seems well with what they are asking for. But then they show you their designs, and you think to yourself, who the hell made these?
The designs look decent, modern even. However, you think about the platform you are using, and you know that this application will stick out like a sore thumb. It has no external consistency with the other apps on the user's device. The components look different, the contrast of colors are different, and the navigation bar is on the top of the screen when it should be at the bottom.

I have been in this kind of situation many times, and I always express my concerns right away. "I think it all looks pretty nice, but I really think for iOS you want to put the toolbar on the bottom, not the top. Usually you put toolbars on the top for Android."

And this is where I crack up on the inside every time: "Oh, I mean, I think it looks ok, and we were actually going to use these designs for both Android and iOS," says the designer.

Let that sink in for a moment.

Would you design a car and a truck with the same standards? No, you wouldn't, and the same thought should occur when designing mobile applications for different platforms. The user expects to see a certain "flavor" of design within your app. Imagine opening your Android app to see the glassy look and thin blue icons of an iOS app. It would take my attention away from the function of the app, and I would be wondering why the app looks like that. Just imagine the confusion for our users who have only used one type of OS for their entire lives.

Not only does the design have a chance to confuse users, but it can also make the development of the application much more challenging. In Android, the default toolbar is set at the top of the screen, with options to either set text for each tab or an icon for each tab. When a developer is tasked with positioning the toolbar at the bottom of the screen, and have each tab contain text and an image, hoops have to be jumped through. The developer needs to add invisible views to push the toolbar down, or add extra layout constraints. Custom views are needed to get the right tab design, and more than often requires some extra touch up to ensure the alignments and font sizes are correct. It is just a hassle, all to create a design effect that goes against the guidelines of the platform.

Telegram is an example of great variation in design for different platforms

Now of course there are always exceptions to these rules; company branding may require different fonts, color schemes, and even UI components. Larger companies can skirt around these issues by even setting standards themselves. However, when designing for any mobile platforms, designers need to take the guidelines set forth by the OS creators into account. I can't stand seeing another single design made for both iOS and Android. There are ways to fit your brand into these guidelines, and even by varying an existing design only slightly from OS to OS, you can create a mobile application that is effective for all users. Remember, as a designer, you are not designing an application that is tailored toward you, or toward your own mobile phone. You are designing for the masses, to make your application accessible to everyone.

Ever find yourself in a similar situation? Even better, have you been the culprit of creating a universal design for the sake of saving time, money, and effort? Let me know!

Creating Beautiful Websites

Learn which front-end and UI frameworks to use when creating modern websites that follow best practices in design.

Written by Aaron Vontell on Sep 12, 2016

Time and time again, I have had friends and colleagues ask me which front-end frameworks and design guidelines to use when making a website. This is an interesting question, as well as a hard one to answer, because everyone has different design tastes. The color scheme you pick may be one person's favorite, but another's worst nightmare.

However, while you cannot tend to the needs and want of every individual who uses your website, you can still follow the best practices for design, use trending styles, and engage a user in a way that they expect to be engaged. Below is a listing of what I believe are the best frameworks for user interface design (with a focus on web technologies), and hopefully they will be of some help to you as well.

Material Design

My most favorite set of design guidelines is without a doubt Material Design, developed by Google (primarily for the mobile OS Android). Material Design has been my go-to set of guidelines for any platform I have had to develop on. There a couple of reasons for this:

Aesthetics
Material Design introduces a set of components that are aesthetically pleasing, piggybacking on the not-so-new trend of simplicity and minimalism. It uses real-world metaphors for displaying objects, which increases the usability and learnability of the UI.

Documentation
I have yet to encounter a set of design guidelines more robust and complete than that for Material Design. The documentation site details every aspect of every component that you would possibly need to put into your website, whether it be dimensions, animations, typography, icons, and more.


Ease of Integration
While the Material Design guidelines are indeed just guidelines, there are many third-party and in-house libraries / frameworks created that implement the designs set forth in the documentation. Two of the best ones I have found are Angular Material and Materialize. As the name might suggest, Angular Material is a front-end framework for AngularJS that implements almost all of the designs, animations, and services outlined in the Material Design specification. Materialize, on the other hand, implements the same set of standards but uses standard CSS and JavaScript (along with SASS for more customization options). Although I personally have not used Materialize much, it is defintely the go-to solution if your website does not use AngularJS.

Bootstrap

If Material Design is not your cup of tea, then another popular option for a set of concrete design standards is those available from Bootstrap. Now, Bootstrap is not a set of design guidelines per se, but it is a library that provides many components that follow a specific theme and feel. Bootstrap also provides their own icons, a very useful gridding system for responsiveness, and it is entirely written in standard CSS and JavaScript. On top of that, there are hundreds of free templates that can be used to get an easy start to your website.

However, in my professional opinion, I would try to stay away from Bootstrap. Now, Bootstrap is a fantastic tool for the early stages of your website. The problem is that literally thousands of websites use Bootstrap. When someone visits your site, there is a good chance that it will look exactly like all of the other sites out there. If you want to design a unique site that grabs a user's attention, then pick a different framework to go with. If anything, I would use the awesome gridding system that Bootstrap has to offer, and then design my own components to give my websites their own unique theme.

Conclusion

Finding the right design for your website is a decision that is entirely up to your own personal likes and dislikes. You may pick and choose how to style your website, but the key is to make sure that the user experiences two things:

  • External Consistency: When the user sees your site for the first time, they should know how to use it through their experience with similar sites. Your styling should not be so crazy as to confuse the user what a certain component does (i.e. don't style a checkbox to look like a radio button).
  • Aesthetics: Ultimately, how your site looks and the way it is organized says a lot about the content of the site, as well as a lot about who you are as a person. No matter what design you go with, have a family member or friend look it over to ensure that it is aethetically pleasing. This will draw your visitors back to your site again and again.

Do you have a favorite design library or set of guidelines? Let me know!
