Basic complexity

  • P = Problems solvable in polynomal time \(O(n^c)\)
    • Ex: Shortest path algorithm
  • NP = Problems whose solutions can be checked in polynomial time (non-deterministic polynomial time)
  • NP-Hard = Problems that are at least as hard as any NP problem
  • NP-Complete = Problems in both NP and NP-hard
    • * For the sake of this class, we assume P ≠ NP and therefore NP - Hard is not in P.

Classifications of complexity classes

Interval scheduling

Interval scheduling is applied when one is looking to arrange a schedule for classes, jobs, tasks, and more. It is usually defined in the following way:

  • There is a single resource (human, CPU, etc...)
  • There are \(n\) requests/classes/jobs, each which runs in an interval \(r_i = [a_i, b_i]\)
  • Requests \(r_i\) and \(r_j\) are compatible if \(r_i \cap r_j = \oslash\)
  • Goal: Find the maximum number of mutually compatible requests.

In our first attempt, we will claim that a greedy algorithm is good. The greedy algorithm will be applied using the following steps:

  1. Look at inputs one at a time, and make a very local choice about it.
  2. Use a simple rule to pick an interval \(r_i\). We put this \(r_i\) into the solution, and discard any other intervals that are incompatible with \(r_i\).
  3. Repeat until we have looked at every interval left in the set of given intervals.

This solution seems to work, but what is the problem? The problem is located in step 2, where we use a 'simple rule' to choose our interval. However, what is this rule? It turns out that defining the rule to pick these intervals is not so trivial. Here are some ideas that might work:

  • Pick the smallest interval first
  • Select the interval with the smallest number of incompatibilities
  • Select the interval with the smallest starting time, or the smallest \(a_i\)
  • Select the interval with the earliest finish time, or the smallest \(b_i\)

It turns out that the last option gives us the best and optimal result, while the others do not (it is quite easy to find counter examples for each case). We can prove this through a "cut and paste" method, where we transform any solution into a greedy solution with no loss in quality.

Let's suppose we have a greedy solution and an optimal solution that has been given to us. The first difference between the two is at interval \(r_i\), where the optimal solution is indicated by \(r'_i\). However, we can claim that \(a'_3 > b_{i-1}\) and \(b'_i > b_i\)

, since the first case would suggest an incompatible schedule and the second case would be solved by our greedy algorithm picking the interval that ended sooner. Therefore, we see that our greedy solution in optimal.

While the greedy algorithm gave optimal solution in this situation, it does not in other cases. In the case of weighted scheduling where we want to maximize the weight, dynamic programming works better. Even worst, when doing Job Interval Scheduling where each interval has multiple possibilities for locations, the problem turns out to be NP-Hard. As you can see, today's messages at the top of the page is very much true.
